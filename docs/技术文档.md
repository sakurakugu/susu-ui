# Susu UI 技术设计

## 目标

Susu UI 是一个个人前端组件库，定位类似 Element Plus。默认优先支持 Vue 3，同时内置主题切换和深色模式能力。

React 和 React Native 可以在后续阶段支持，但不要一开始把不同平台的组件实现混在一起。推荐做法是共享设计令牌和非 UI 逻辑，组件渲染按平台分别实现。

## 包结构策略

项目使用 monorepo 管理，让共享基础能力可以独立演进。

```txt
susu-ui/
  packages/
    tokens/       设计令牌，跨平台共享。
    theme/        Web 端 CSS 变量、主题样式和运行时切换方法。
    vue/          Vue 3 组件库，当前默认主包。
    react/        后续 React 组件库。
    rn/           后续 React Native 组件库。
    shared/       后续共享工具和无框架逻辑。
  playground/     本地 Vue 调试示例。
  docs/           技术文档，后续可扩展为文档站。
```

第一阶段只落地：

- `@susu-ui/tokens`
- `@susu-ui/theme`
- `@susu-ui/vue`
- `playground`

React 和 React Native 建议等 token 模型、主题模型、Vue 组件 API 稳定后再追加。

## 设计令牌

设计令牌使用与框架无关的 TypeScript 对象描述。命名应优先表达语义，而不是绑定具体组件。

推荐分类：

- 颜色：`primary`、`success`、`warning`、`danger`、`bg`、`text`、`border`
- 圆角：`sm`、`md`、`lg`、`round`
- 间距：`xs`、`sm`、`md`、`lg`、`xl`
- 字体：`sizeSm`、`sizeMd`、`sizeLg`、`lineHeight`
- 阴影：`sm`、`md`、`lg`

组件内部不要硬编码颜色。Web 组件使用语义化 CSS 变量，React Native 组件使用主题对象。

## Web 主题模型

Web 端主题使用 CSS 变量实现，并把明暗模式和主题色分成两条轴：

- 明暗模式：通过 `data-su-theme` 控制，支持 `light`、`dark`、`system`。
- 主题色：通过 `--su-theme-hue` 控制，使用 OKLCH 派生主色、悬停色和激活色。

```css
:root {
  --su-theme-hue: 215;
  --su-color-primary: oklch(0.62 0.16 var(--su-theme-hue));
  --su-color-primary-hover: oklch(0.68 0.15 var(--su-theme-hue));
}

:root,
[data-su-theme='light'] {
  --su-color-bg: #ffffff;
  --su-color-text: #1f2329;
}

[data-su-theme='dark'] {
  --su-color-bg: #141414;
  --su-color-text: #f0f0f0;
}
```

运行时切换主题时，只更新根节点属性和 CSS 变量：

```ts
setThemeMode('dark')
setThemeMode('light')
setThemePreset('pink')
setThemeHue(150)
```

这种方式不依赖 Vue 组件状态，切换成本低，也方便未来给 React Web 复用。

## Vue 组件模型

每个组件独立维护 Vue 实现和样式入口：

```txt
packages/vue/src/components/button/
  Button.vue
  index.ts
```

组件样式使用稳定的 `su-` 类名前缀，并只消费 CSS 变量：

```css
.su-button {
  color: var(--su-color-primary-text);
  background: var(--su-color-primary);
  border-radius: var(--su-radius-md);
}
```

Vue 组件需要同时支持按需导入和全局插件安装：

```ts
import { SuButton } from '@susu-ui/vue'
```

```ts
import { createApp } from 'vue'
import { SusuUI } from '@susu-ui/vue'
import '@susu-ui/theme/style.css'

createApp(App).use(SusuUI).mount('#app')
```

## React 支持策略

React 应该作为独立包实现：

```txt
packages/react/
```

React 可以共享：

- `@susu-ui/tokens`
- `@susu-ui/theme` 的 Web CSS 变量
- `@susu-ui/shared` 的无框架工具函数

不要用 React 包去包裹 Vue 组件。React 组件应该用原生 React 写法实现，但尽量复用相同的 class 命名、CSS 变量和 token 语义。

## React Native 支持策略

React Native 应该作为独立包实现：

```txt
packages/rn/
```

React Native 没有 DOM，也不能直接使用 CSS 变量。它应该直接消费 token 对象，并通过 Provider 下发主题：

```tsx
<SusuThemeProvider theme={lightTheme}>
  <App />
</SusuThemeProvider>
```

Web 和 React Native 的共享契约是 token 结构，不是 CSS 实现。

## 构建策略

推荐工具：

- `pnpm` workspace
- `vite` 负责 playground 和 Vue 库构建
- `vue-tsc` 负责 Vue 类型检查和声明文件
- 后续可用 `tsup` 构建非 Vue TypeScript 包
- 后续可用 `vitest` 做单元测试
- 后续可用 `vitepress` 或 Storybook 做文档站

当前初始化阶段优先保证：

1. token 命名稳定。
2. CSS 变量和运行时主题切换可用。
3. Vue 组件 API 面足够小。
4. 本地 playground 可以做视觉验证。

## 发布流程

项目使用 Changesets 管理包版本和发布记录。发布前先确认 npm 登录状态：

去 npm 官网创建 Granular Access Token，并给需要发布的包开启写权限。如果账号开启了 2FA，需要在 token 权限中勾选绕过 2FA 发布校验，否则非交互终端发布时可能出现 `ERR_PNPM_OTP_NON_INTERACTIVE`。

本地推荐优先把 token 放到当前终端环境变量中，避免写入仓库文件：

```bash
export NPM_TOKEN="npm_xxx"
npm whoami --registry https://registry.npmjs.org/
```

PowerShell 使用：

```powershell
$env:NPM_TOKEN="npm_xxx"
npm whoami --registry https://registry.npmjs.org/
```

写入用户环境变量:

```powershell
[Environment]::SetEnvironmentVariable("NPM_TOKEN", "npm_xxx", "User")
```

如果要写入 npm 当前用户配置，可以使用 npm config。这个配置会写入用户目录下的 `.npmrc`，不要写入仓库根目录：

```bash
npm config set registry "https://registry.npmjs.org/"
npm config set //registry.npmjs.org/:_authToken "npm_xxx"
npm whoami --registry https://registry.npmjs.org/
```

如果确实需要项目级 `.npmrc`，只能写环境变量占位符，不能提交明文 token：

```ini
registry=https://registry.npmjs.org/
//registry.npmjs.org/:_authToken=${NPM_TOKEN}
```

CI 发布时，把 `NPM_TOKEN` 放到 CI Secret 中，再在发布任务里生成临时 `.npmrc`：

```bash
echo "registry=https://registry.npmjs.org/" > .npmrc
echo "//registry.npmjs.org/:_authToken=${NPM_TOKEN}" >> .npmrc
npm whoami --registry https://registry.npmjs.org/
```

发布版本前，需要确认 `.changeset/*.md` 中的包名和升级类型符合预期。升级类型含义如下：

- `patch`：修复问题或小改动，例如 `0.1.0` 升到 `0.1.1`。
- `minor`：新增组件或功能，例如 `0.1.0` 升到 `0.2.0`。
- `major`：破坏性变更，例如 `0.1.0` 升到 `1.0.0`。

例如发布 `@susu-ui/vue@0.2.0`，需要把对应 changeset 写成 `minor`：

```md
---
'@susu-ui/vue': minor
---

创建 susu-ui。
```

如果只声明 `@susu-ui/vue`，则只会发布 Vue 组件库；`@susu-ui/theme` 和 `@susu-ui/tokens` 会保持原版本。若三个包都要发布 `0.2.0`，changeset 需要同时声明：

```md
---
'@susu-ui/vue': minor
'@susu-ui/theme': minor
'@susu-ui/tokens': minor
---

发布 0.2.0 版本。
```

确认 changeset 后，生成版本号和更新日志：

```bash
pnpm version-packages
```

发布前运行完整检查：

```bash
pnpm check
```

检查通过后开始发布到 npm:

> 要保持git中没有暂存的内容，Changesets 2.31.0 没正确处理 pnpm 这个错误

```bash
git status
git add .
```

```bash
git commit -m "发布: 0.2.0"
```

```bash
pnpm release
git push
git push --tags
```

如果 `pnpm release` 中的 `changeset publish` 已确认 token 有发布权限，但仍然因为 `ERR_PNPM_OTP_NON_INTERACTIVE` 失败，可以先确认构建和导出检查已通过，再对未发布的包使用 npm CLI 直接发布：

```bash
pnpm build
pnpm check:exports
cd packages/tokens
npm publish --access public --registry=https://registry.npmjs.org/
cd ../vue
npm publish --access public --registry=https://registry.npmjs.org/
```

发布后确认版本：

```bash
npm view @susu-ui/tokens@0.2.0 version --registry=https://registry.npmjs.org/
npm view @susu-ui/theme@0.2.0 version --registry=https://registry.npmjs.org/
npm view @susu-ui/vue@0.2.0 version --registry=https://registry.npmjs.org/
```
